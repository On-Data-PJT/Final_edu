# Contributing

## 브랜치 규칙

- 기능 개발: `feat/<short-topic>`
- 버그 수정: `fix/<short-topic>`
- 문서 작업: `docs/<short-topic>`

예시:

- `feat/team-onboarding`
- `fix/readme-command`
- `docs/contest-notes`

## 작업 흐름

1. `main` 최신 상태를 기준으로 작업 브랜치를 만듭니다.
2. 로컬에서 `uv sync` 후 작업합니다.
3. 변경 사항을 확인하고 Pull Request를 생성합니다.
4. 최소 1회 리뷰 후 `main`에 머지합니다.

## Pull Request 최소 기준

- 변경 목적이 제목만 봐도 드러나야 합니다.
- 본문에 작업 이유와 영향 범위를 짧게 적습니다.
- 실행 또는 확인한 명령을 적습니다.
- 미해결 이슈가 있으면 본문에 남깁니다.

## 커밋 원칙

- 작은 단위로 나눠 커밋합니다.
- 의미 없는 중간 결과물은 정리 후 올립니다.
- 비밀정보, 개인 설정, 로컬 캐시는 커밋하지 않습니다.

## 공모전 주의사항

- 마감일은 `2026-04-13`입니다.
- 마감 이후 커밋은 부정행위로 해석될 수 있으므로, 제출 직전에는 머지와 배포 일정을 엄격히 관리합니다.

## 문서 언어

- 프로젝트 문서는 기본적으로 한국어로 작성합니다.
- 코드, 패키지명, 환경 변수명은 영어를 사용합니다.
